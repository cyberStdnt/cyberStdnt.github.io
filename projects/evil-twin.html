Evil Twin Portal · ESP32-S3

Firmware autonome pour ESP32-S3. Monté sur un module à 5 €, il se comporte comme un rogue Access Point avec panneau d'administration web, scanner Wi-Fi intégré, injection de trames deauthentication 802.11 sans saut de canal, et portail captif qui valide en temps réel le mot de passe saisi contre le vrai réseau cible. Étude pédagogique construite pour documenter les failles intrinsèques de WPA2-Personal et argumenter la migration vers WPA3-SAE + 802.11w.

Standalone firmware for ESP32-S3. Running on a €5 module, it behaves as a rogue Access Point with a web admin panel, built-in Wi-Fi scanner, 802.11 deauthentication frame injection without channel hopping, and a captive portal that validates in real time the submitted password against the true target network. Educational study built to document WPA2-Personal's structural flaws and argue the migration to WPA3-SAE + 802.11w.

Pourquoi un ESP32-S3 plutôt qu'un PC Linux

Why ESP32-S3 over a Linux laptop

Les tutoriels grand public font tourner ce type d'attaque sous Kali avec hostapd + dnsmasq + aircrack-ng sur deux cartes USB. Ça marche, mais ça masque la réalité : un ESP32-S3 à 5 €, alimenté sur batterie, tient dans la paume. Le passage au monde embarqué change totalement la perspective défensive — un module aussi petit, autonome, qui diffuse un portail captif crédible, ce n'est plus un scénario théorique, c'est une menace plausible dans un café, un aéroport, un hall de conférence. L'objectif pédagogique du projet est précisément de rendre tangible cette menace pour mieux justifier les mitigations côté infrastructure et côté utilisateur.

Mainstream tutorials run this attack on Kali with hostapd + dnsmasq + aircrack-ng on two USB NICs. It works, but it hides reality: a €5 ESP32-S3, battery-powered, palm-sized. Moving to the embedded side completely shifts the defensive perspective — a module this small, standalone, serving a credible captive portal, stops being theoretical and becomes a plausible threat in a café, airport, or conference hall. The educational goal of this project is precisely to make that threat tangible to better justify mitigations on both the infrastructure and user sides.

Les 4 pages servies

The 4 served pages

  • /admin — panneau de contrôle (thème sombre) : liste des SSIDs scannés avec RSSI / canal / BSSID, sélection de la cible, boutons Deauth / EvilTwin, affichage du mot de passe capturé.
  • /login — portail captif (thème clair façon page système) : pretexte « Mise à jour du firmware échouée — confirmez votre mot de passe Wi-Fi pour restaurer l'ancienne version ». Le pretexte social est volontaire — c'est la vraie vulnérabilité, pas la partie technique.
  • /portal — écran d'attente avec spinner et barre de progression animée (~16 s), le temps que le firmware tente la connexion au vrai réseau.
  • /success — confirmation que le « firmware a été restauré » : la victime repart sans soupçon, l'AP clone reste ouvert sur le canal admin.
  • /admin — control panel (dark theme): list of scanned SSIDs with RSSI / channel / BSSID, target selection, Deauth / EvilTwin toggle buttons, captured password display.
  • /login — captive portal (light theme, system-page feel): social pretext "Firmware update failed — confirm your Wi-Fi password to roll back to the previous version". The social pretext is deliberate — that's the real vulnerability, not the technical bit.
  • /portal — waiting screen with animated spinner and progress bar (~16 s), while the firmware attempts connection to the real network.
  • /success — confirmation that the "firmware was restored": the victim leaves without suspicion, the clone AP stays open on the admin channel.

Points techniques saillants

Technical highlights

Contournement du filtre ESP-IDF 5.x sur les trames de management

Bypassing the ESP-IDF 5.x management-frame filter

Sur arduino-esp32 ≥ 3.x (basé sur IDF 5.x), l'API publique esp_wifi_80211_tx() rejette les frame types 0xC0 (Deauthentication) et 0xA0 (Disassociation) avec un unsupport frame type. La fonction ieee80211_raw_frame_sanity_check() qui était weak symbol dans les versions précédentes — et qu'on pouvait surcharger pour faire sauter le filtre — ne l'est plus. Solution retenue : descendre à l'API interne esp_wifi_internal_tx(), qui ne passe par aucun filtre de type de trame.

On arduino-esp32 ≥ 3.x (IDF 5.x-based), the public esp_wifi_80211_tx() API rejects frame types 0xC0 (Deauthentication) and 0xA0 (Disassociation) with unsupport frame type. The ieee80211_raw_frame_sanity_check() function — a weak symbol in older versions, overridable to disable the filter — no longer is. Chosen fix: drop down to the internal API esp_wifi_internal_tx(), which bypasses the frame-type filter entirely. 

// Trame deauth broadcast 802.11 (26 octets)
uint8_t pkt[26] = {
  0xC0, 0x00,                         // Frame Control : Deauthentication
  0x00, 0x00,                         // Duration
  0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF, // Destination : broadcast
  0x00, 0x00, 0x00, 0x00, 0x00, 0x00, // Source  : BSSID cible (spoofé)
  0x00, 0x00, 0x00, 0x00, 0x00, 0x00, // BSSID   : BSSID cible
  0x00, 0x00,                         // Sequence
  0x07, 0x00                          // Reason 7 : Class 3 frame from non-auth STA
};
memcpy(&pkt[10], target.bssid, 6);
memcpy(&pkt[16], target.bssid, 6);
esp_wifi_internal_tx(WIFI_IF_AP, pkt, sizeof(pkt));

Pas de saut de canal

No channel hopping

L'ESP32-S3 n'a qu'une seule radio. Si on fait esp_wifi_set_channel() pour injecter sur le canal de la cible pendant qu'un client est connecté à notre AP admin, on déconnecte ce client. Le design choisi ici contourne le problème : l'AP clone est démarré directement sur target.ch via softAP(ssid, "", channel). L'injection deauth se fait sur le canal courant de l'AP — aucun saut nécessaire, l'attaquant reste connecté à son panneau admin pendant toute l'opération.

The ESP32-S3 has a single radio. Calling esp_wifi_set_channel() to inject on the target's channel while a client is connected to our admin AP would disconnect that client. The chosen design sidesteps this: the clone AP is started directly on target.ch via softAP(ssid, "", channel). Deauth injection happens on the AP's current channel — no hopping required, the attacker stays connected to the admin panel throughout.

Validation du mot de passe en direct

Live password validation

Quand la victime soumet /login, le firmware passe en mode STA et tente immédiatement WiFi.begin(target.ssid, triedPwd, target.ch, target.bssid). Si WiFi.status() == WL_CONNECTED → le mot de passe est bon, on l'affiche dans le panel admin et la victime est redirigée vers /success. Sinon → retour vers /login et la deauth reprend pour forcer une nouvelle tentative. Aucun mot de passe bidon n'est capturé — la victime ne peut pas s'en sortir avec un faux.

When the victim submits /login, the firmware switches to STA mode and immediately attempts WiFi.begin(target.ssid, triedPwd, target.ch, target.bssid). If WiFi.status() == WL_CONNECTED → password is valid, displayed in the admin panel, victim redirected to /success. Otherwise → back to /login and deauth resumes, forcing another attempt. No bogus passwords get captured — the victim can't escape with a fake one.

Contre-mesures — la partie importante

Mitigations — the important part

Côté utilisateur

User side

  • Supprimer les profils Wi-Fi publics / ouverts non utilisés — coupe l'auto-join sur SSID connu.
  • Ne jamais saisir de mot de passe sur une page servie en HTTP. Cadenas absent, HSTS non actif → on s'arrête.
  • VPN systématique sur Wi-Fi non maîtrisé. La gratuité d'un service Wi-Fi n'est pas un critère de confiance.
  • Sur iOS / Android : désactiver la reconnexion automatique aux réseaux ouverts.
  • Delete unused open / public Wi-Fi profiles — breaks auto-join on known SSIDs.
  • Never type a password on an HTTP page. No padlock, no active HSTS → stop.
  • Always-on VPN on any Wi-Fi you don't control. Free ≠ trusted.
  • iOS / Android: disable auto-join to open networks.

Côté infrastructure

Infrastructure side

  • WPA3-SAE — le handshake Dragonfly résiste aux attaques offline classiques. Migration en priorité.
  • 802.11w (PMF) en mode required — les trames de management sont signées, les deauth deviennent inutilisables. C'est ce qui casse précisément ce projet.
  • WIDS (Kismet, Aruba RFProtect, Cisco WIPS) qui alerte sur un ESSID dupliqué avec un BSSID inattendu.
  • En environnement pro — 802.1X / WPA2-Enterprise avec vérification stricte du certificat serveur côté client. Ce paramètre est mal configuré dans 90 % des déploiements ; à auditer.
  • WPA3-SAE — the Dragonfly handshake resists classic offline attacks. Top migration priority.
  • 802.11w (PMF) in required mode — management frames get signed, deauth becomes unusable. This is exactly what breaks the current project.
  • WIDS (Kismet, Aruba RFProtect, Cisco WIPS) that alerts on duplicate ESSID under unexpected BSSID.
  • In enterprise — 802.1X / WPA2-Enterprise with strict server-certificate validation on the client. Misconfigured in 90 % of deployments; needs auditing.
